# Размещение на GitHub Pages

---

## Сначала — честно о том, что придётся изменить

GitHub Pages раздаёт только статические файлы. Серверный код там не выполняется.
Из-за этого три части проекта работают иначе:

| Что было на PHP-хостинге | Что будет на GitHub Pages |
|---|---|
| Панель `api/admin.php` с тремя вкладками | Панели нет. Альбомы собираются из папок в `img` автоматически |
| Заявки копятся в базе, видны в панели | Заявки уходят на почту через сервис-приёмник |
| Фото загружаются кнопкой в панели | Фото кладутся в папку и пушатся — код трогать не нужно |

Взамен вы получаете бесплатный HTTPS, свой домен, отсутствие рекламы,
резервную копию всего сайта в истории коммитов и хостинг, который не упадёт.

**Одно важное следствие.** На бесплатном тарифе GitHub Pages работает только
из публичного репозитория. Это значит, что все исходники и **все фотографии
может скачать кто угодно** — не через сайт, так напрямую из репозитория.
Для портфолио это обычно приемлемо, но если снимки коммерческие и вы
не хотите отдавать оригиналы — либо кладите уменьшенные версии,
либо берите тариф GitHub Pro, где Pages работает и с закрытым репозиторием.

Лимиты: сайт до 1 ГБ, около 100 ГБ трафика в месяц. Для портфолио с запасом,
но фотографии — тяжёлый контент, следите за весом.

И ещё: правила GitHub Pages прямо запрещают использовать сервис
для интернет-магазинов и коммерческих площадок. Портфолио с формой связи
под этот запрет не попадает, а вот продажу съёмок прямо на сайте
лучше туда не выносить.

---

## Шаг 1. Репозиторий

1. Создайте на GitHub новый **публичный** репозиторий.
2. Загрузите в него содержимое папки `portfolio` — саму папку не переносите,
   `index.html` должен лежать в корне репозитория.

Через сайт: кнопка **Add file → Upload files**, перетащите всё разом.
Через командную строку:

```bash
cd portfolio
git init
git add .
git commit -m "Портфолио"
git branch -M main
git remote add origin https://github.com/ВАШ-ЛОГИН/ИМЯ-РЕПОЗИТОРИЯ.git
git push -u origin main
```

Файлы, которые на GitHub Pages не нужны, но и не мешают:
папка `api`, `.htaccess`, `start.sh`, `start.bat`. Можете удалить.

---

## Шаг 2. Включение Pages

**Settings → Pages → Build and deployment → Source: GitHub Actions**

Именно Actions, а не «Deploy from a branch» — сборщик альбомов работает
как раз на этом этапе.

Больше ничего настраивать не нужно: файл `.github/workflows/pages.yml`
уже лежит в архиве. При первом пуше запустится сборка, её видно
во вкладке **Actions**. Через минуту-две адрес сайта появится
в **Settings → Pages**. Он будет вида:

```
https://ваш-логин.github.io/имя-репозитория/
```

Что происходит при каждом пуше:

1. Скрипт `tools/build_albums.py` просматривает папку `img`
2. Собирает `data/albums.json` — список альбомов и фоновых фото
3. Сайт публикуется

Если сборка упала — откройте Actions и посмотрите красный шаг,
там написана причина.

## Шаг 3. Фотографии

Ничего в коде править не нужно. Раскладываете файлы — при пуше
сборщик сам находит их и обновляет сайт.

**Фоновые снимки** кладите прямо в `img/` с такими именами:

```
img/cover.jpg        фон заставки
img/about-bg.jpg     фон раздела «О себе»
img/portrait.jpg     портрет автора
img/contact.jpg      фото в блоке «Связаться»
```

Расширение любое: jpg, png, webp, avif.

**Альбомы** — папка на альбом внутри `img/albums/`:

```
img/albums/01-портреты/photo-1.jpg
img/albums/01-портреты/photo-2.jpg
img/albums/02-репортаж/cover.jpg
img/albums/02-репортаж/street-1.jpg
```

Правила простые:

- **Название альбома** берётся из имени папки: цифры в начале отбрасываются,
  дефисы становятся пробелами. `01-художественная-съёмка` превратится
  в «Художественная съёмка».
- **Нужно точное название** — положите в папку файл `title.txt`
  и напишите его первой строкой.
- **Порядок альбомов** задают цифры в начале имени папки.
- **Обложка** — файл с именем `cover.jpg`. Нет такого — берётся первое фото.
- **Порядок фотографий** — по именам файлов, причём `photo-2` идёт
  перед `photo-10`, а не наоборот.

Имена латиницей надёжнее. Кириллица работает — я проверял, — но некоторые
программы обращаются с ней по-разному, и лишний источник проблем ни к чему.

Добавить фото через сайт GitHub: зайдите в нужную папку,
**Add file → Upload files**, перетащите снимки, Commit changes.
Через минуту они уже на сайте.

**Подготовка снимков.** Обложки под 21:9 шириной до 2000 px, формат WebP,
вес до 300 КБ. При лимите в 1 ГБ это около трёх тысяч фотографий.

## Шаг 4. Форма обратной связи

Заявки будет принимать сторонний сервис и пересылать вам на почту.
Бесплатных вариантов два, оба настраиваются за пять минут.

### Web3Forms — без регистрации

1. Зайдите на web3forms.com, введите свою почту, получите ключ доступа.
2. Откройте `index.html`, найдите блок `CONFIG` в самом низу файла:

```js
formTo: 'web3forms',
web3formsKey: 'сюда-вставить-ключ',
```

### Formspree — с личным кабинетом и историей заявок

1. Зарегистрируйтесь на formspree.io, создайте форму, скопируйте её адрес.
2. В блоке `CONFIG`:

```js
formTo: 'formspree',
formspreeUrl: 'https://formspree.io/f/xxxxxxx',
```

Бесплатный тариф ограничен по числу заявок в месяц — посмотрите
актуальные условия на сайте сервиса, они меняются.

**Про безопасность.** Ключ сервиса лежит прямо в коде страницы, и это
нормально: он рассчитан на публичное использование и позволяет только
отправлять заявки вам, а не читать их. Скрытое поле-ловушка от ботов
в форме осталось и передаётся обоим сервисам.

**Про персональные данные.** Заявки теперь идут через чужой сервер.
Для сайта, ориентированного на российскую аудиторию, это стоит отразить
в политике конфиденциальности. Скажите, если нужно — подготовлю текст
и галочку согласия рядом с кнопкой.

## Шаг 5. Свой домен

**Settings → Pages → Custom domain**, впишите домен и сохраните.
GitHub создаст в репозитории файл `CNAME`.

У регистратора домена пропишите записи:

- поддомен вроде `www.example.ru` → запись CNAME на `ваш-логин.github.io`
- корневой домен `example.ru` → четыре записи A на адреса
  `185.199.108.153`, `185.199.109.153`, `185.199.110.153`, `185.199.111.153`

Через несколько часов вернитесь и поставьте галочку **Enforce HTTPS**.
Сертификат бесплатный и продлевается сам.

## Проверка

1. Откройте адрес сайта — должна появиться заставка на весь экран.
2. Пролистайте до альбомов, откройте один — работает просмотр, стрелки, Esc.
3. Отправьте тестовую заявку и проверьте почту.

## Если что-то не так

**Открывается пустая страница или 404.**
Подождите пару минут после первого включения. Проверьте, что `index.html`
лежит именно в корне репозитория, а не внутри папки `portfolio`.

**Сайт открылся, но нет стилей и картинок.**
Скорее всего, где-то остался путь, начинающийся со слэша. В этой сборке
все пути относительные — если правили сами, уберите ведущий `/`:
нужно `img/cover.jpg`, а не `/img/cover.jpg`.

**Альбомы показывают заглушки вместо фотографий.**
Либо сломан `data/albums.json`, либо пути к файлам не совпадают
с реальными именами. Имена чувствительны к регистру:
`Art-01.JPG` и `art-01.jpg` — разные файлы.

**Форма пишет «Отправить не получилось».**
Не заполнен ключ в блоке `CONFIG` либо в нём опечатка.
Откройте консоль браузера (F12) — там будет видна причина.

**Новые фотографии не появились.**
Загляните во вкладку Actions — идёт ли сборка и не упала ли она.
Частая причина: файл лежит не в папке внутри `img/albums`,
а прямо в `img`. Проверьте расширение — поддерживаются
jpg, jpeg, png, webp, avif, gif.

**Изменения не появляются на сайте.**
Публикация занимает до пары минут. Загляните во вкладку Actions —
там видно, идёт ли сборка. И обновите страницу с очисткой кеша: Ctrl+F5.

## Если захочется вернуть панель управления

Вариант остаётся: положите сайт на любой хостинг с PHP, а в блоке `CONFIG`
поменяйте две строки:

```js
albumsFrom: 'php',
formTo: 'php',
```

Все файлы для этого уже лежат в папке `api` — удалять её не обязательно,
на GitHub Pages она просто не выполняется и никому не мешает.
